decltype() and std::declval<T>()

decltype()

Roughly speaking, it gives us the type of an expression or entity.

  • If what passed to decltype is the name of a variable or function or a member of an object, then the result is the type of whatever this refers to, this includes reference, const and volatile specifiers.
const int i = 0; // decltype(i) is const int
bool f(const Widget& w); // decltype(w) is const Widget&
                         // decltype(f) is bool(const Widget&)
struct Point {
  int x, y; // decltype(Point::x) is int
};          // decltype(Point::y) is int
Widget w;   // decltype(w) is Widget
if (f(w)) … // decltype(f(w)) is bool
  • An exception to above rule is C++17’s structured binding: If the name we pass to decltype is a variable defined in structured binding, then the result is the type of the bound-to element.
    std::pair<int volatile &&, double&> f(int);
auto const& [a, b] = f(22); // type of a is "int volatile&&", not "int const volatile&"
  • If what passed to decltype is not just a name or member access expression, the resulting type depends on the value category of the expression. Given the type of the expression e is E, then decltype(e) is
    • E, if e is an prvalue,
    • E&, if e is an lvalue
    • E&&, if e is a xvalue
int x = 0; // decltype(x) is int.
           // decltype((x)) is int&, because C++ defines the expression (x) to be an lvalue

In C++11, one primary use for decltype is declaring function templates where the function’s return type depends on its parameter types. C++11 permits return types for single-statement lambdas to be deduced, and C++14 extends this to both all lambdas and all functions, including those with multiple statements.

template<typename Container, typename Index> 
auto authAndAccess(Container& c, Index i) 
-> decltype(c[i]) // needed for C++11, but can be omitted in C++14
{
  ...
  return c[i]; // c[i] type is T&
}

In above example, operator[] for most containers-of-T returns a T&, during auto return type's template type deduction, the reference of an initializing expression is ignored. Thus, auto return type is deduced as int. To retain the same type of expression c[i], we need decltype(auto).

decltype(auto)

auto deduction rule ignores specifiers including reference and const. decltype(auto) allows auto declarations to use the decltype rules on the given expression.

Widget w;
const Widget& cw = w;
auto myWidget1 = cw;           // auto type deduction: myWidget1's type is Widget
decltype(auto) myWidget2 = cw; // decltype type deduction: myWidget2's type is const Widget&

decltype(auto) is primarily useful for deducing the return type of forwarding functions and similar wrappers, where you want the type to exactly “track” some expression you’re invoking.

template<class Fun, class... Args>
decltype(auto) Example(Fun fun, Args&&... args) { 
    return fun(std::forward<Args>(args)...); 
}

std::declval<T>()

  • returns an rvalue reference to type T
    • It returns a "fake" instance (pseudo-instance) of a T object, and at the same time has an rvalue reference, ie, T&&.
  • makes it possible to call member functions without an object
  • makes it possible to call functions that aren't implemented (they only need to be declared)
  • especially useful in decltype expressions.
  • declval can only be used in unevaluated contexts.

We often don't really need the pseudo-instance obtained by declval evaluation directly, but we need to use this pseudo-instance to get the corresponding type description. Therefore, in general, declval is often surrounded by decltype calculations. Trying to get the type is our real goal.

// get result type of calling A::foo() 
using foo_t = decltype(declval<A>().foo());

// get result type of adding objects of type X and object of type Y, including const and references
using deref_t = decltype(declval<X>() + declval<Y>());

declval is commonly used in templates where acceptable template parameters may have no constructor in common, but have the same member function whose return type is needed.

struct Default { int foo() const { return 1; } };
 
struct NonDefault {
    NonDefault() = delete;
    int foo() const { return 1; }
};
 
decltype(Default().foo()) n1 = 1;                   // type of n1 is int
//  decltype(NonDefault().foo()) n2 = n1;           // ERROR: no default constructor
decltype(std::declval<NonDefault>().foo()) n2 = n1; // type of n2 is int

Another example: useful in concepts (C++20) definition

template<typename T>
concept SupportsPushBack = requires(T coll) {
  coll.push_back(std::declval<typename T::value_type>());
};

Using std::declval<T&>()

To get an lvalue reference from declval you use std::declval<T&>() because it would then return T&&& which will collapse to T&.

This is useful if passing the value to a function that takes a non-const reference because passing an rvalue reference would be a compile error.



extern template

https://arne-mertz.de/2019/02/extern-template-reduce-compile-times/

Pro

  • faster build time, because the function only needs to be compiled once

Con

  • the function could be inlined if it is regular template, as the template instantiation happens in the same file. When it is declared as extern template, the function in general won't be inlined as it is defined in another compilation unit.
    • LTO can sometimes solve above issue refer to this video, but you can't rely on that.

Guideline

  • Consider use extern template for a heavliy used expensive function (it won't be inlined anyway) to reduce compilation time.


Fibonacci

Recursive Approach

  • Time Complexity - O(2^N)
  • Space Complexity - O(N) given the function call stack size
   int fib(int N) {
        if(N == 0)  return 0;
        if(N == 1)  return 1;
        return fib(N-1) + fib(N-2);
    }

Dynamic Programming Approach

Use memoization to store perviously computed fibonacci values.

  • Time Complexity - O(N)
  • Space Complexity - O(N)
   int fib(int N) {
        if(N < 2)
            return N;
        int memo[N+1];
        memo[0] = 0;
        memo[1] = 1;
        for(int i=2; i<=N; i++)
            memo[i] = memo[i-1] + memo[i-2];
        return memo[N];
    }

Template

Things to note:

  • Overflow, int64_t supports about fib<92>
  • If we pass a negative number, your compiler will protect you from creating literally infinite templates (GCC 7.2.1 allowed 900 to be generated, use -ftemplate-depth= to change it)
template<int n>
struct fibonacci
{
  //static_assert(n >= 0, "Error: fib can't be called with a negative integer");
  static constexpr int64_t value = fibonacci<n-1>::value + fibonacci<n-2>::value;
};
template<>
struct fibonacci<0>
{
  static constexpr int64_t value = 0;
};
template<>
struct fibonacci<1>
{
  static constexpr int64_t value = 1;
};

int main()
{
    fibonacci<42>::value;
    return 0;
}
