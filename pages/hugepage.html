Huge pages

Huge pages means big page size (either 2M or 1G), supported in recent Intel and AMD CPUs. It means TLB can hold large address spaces, which in turns means less TLB misses, and less page table lookups.

Linux official doc - 5 parts: https://lwn.net/Articles/374424/

64位的x86-64系统，huge page的大小是2MB或者1GB，初始数目由启动时的"vm.nr_hugepages" 内核参数确定。

  • 对于2MB的huge page，运行过程中可通过"/proc/sys/vm/nr_hugepages"修改。
  • 对于1GB的huge page，就只能在启动时分配（且分配后不能释放），而不支持在运行时修改。

Huge page fault behavior

Since 2.6.18, huge pages were faulted the same as normal mappings when the page was first referenced. To guarantee that faults would succeed, huge pages were reserved at the time when mapping is created. This applies to both shared and private mapping (since 2.6.29).

  • Shared mappings are guaranteed to successfully fault regardless of what process accesses the mapping.
  • For private mappings, only the process that creates the mapping mmap() is guaranteed to successfully fault. When that process fork()s, two processes are now accessing the same pages. If the child performs COW, an attempt will be made to allocate a new page. If it succeeds, the fault successfully completes. If the fault fails, the child gets terminated with a message logged to the kernel log noting that there were insufficient huge pages.

Use huge pages for anonymous mmap()

As of kernel 2.6.32, support is available that allows anonymous mappings to be created backed by huge pages with mmap() by specifying the flags MAP_ANONYMOUS|MAP_HUGETLB. These mappings can be private or shared.

Use huge pages for file-backed mmap()

For the creation of shared or private mappings, Linux provides a RAM-based filesystem called "hugetlbfs." Every file on this filesystem is backed by huge pages and is accessed with mmap() or read().

Using huge pages for malloc

Glibc defines a __morecore hook that is is called when the heap size needs to be increased; libhugetlbfs uses this hook to create regions of memory backed by huge pages.

We don't need to make any modifications to use huge pages for malloc. You can use huge pages for malloc by one of the following options:

Option 1: using environment variables

  • LD_PRELOAD=libhugetlbfs.so
  • LD_LIBRARY_PATH to directly containing libhugetlbfs.so which will generally be either in /usr/lib or /usr/lib64. You can set it to /usr/lib:/usr/lib64: in general.
  • HUGETLB_MORECORE=yes

Option 2: using hugectl (Recommended)

  • You simply need to run your binary with prefix "hugectl --heap". This will use the default page size for pages used for malloc. In case you have multiple options for huge pages and want to use non-default page size, you can do so by prefixing your binary with "hugectl --heap=$PAGE_SIZE" instead.

Using huge pages for shmem

Similar to malloc, you don't need to make any modifications to your binary. You can use huge pages using one of the following options:

Option 1: using environment variables

  • same as enviroment variables used in malloc case.

Option 2: using hugectl (Recommended)

  • Run binary with the prefix "hugectl --shm". Unlike malloc, shmem can only be mapped in default hugepage size.

Using huge pages for text/data/bss section

Linking

  • For this you'll have to relink your binary with "-zcommon-page-size $PAGE_SIZE -zmax-page-size $PAGE_SIZE -lhugetlbfs".

Running

Option 1: using environment variables

  • When you run your binary, you'll have to set HUGETLB_ELFMAP to one of these values:
    • HUGETLB_ELFMAP=R for using huge pages for text section
    • HUGETLB_ELFMAP=RW for using huge pages for text and data section
    • HUGETLB_ELFMAP=W for using huge pages for data section
    • HUGETLB_ELFMAP=N (no hugepages for text/data) Notice that you can either choose to map both data and BSS section or none of them, but we don't have an option to use huge pages for only one of them. Above settings will allow you to use default page size for these sections, if you want to use some other page size you can use following syntax: HUGETLB_ELFMAP=[R[=<pagesize>]:[W[=<pagesize>]]

Option 2: using hugectl (Recommended)

  • Run your binary with prefix "hugectl --text=$PAGE_SIZE --data=$PAGE_SIZE --bss=$PAGE_SIZE". Note that using either --data or --bss option will put both data and bss section in huge pages.

Other things to keep in mind

Are you using mmap syscall with MAP_HUGETLB flag enabled?

  • mmap syscall has a parameter flags and you have enabled MAP_HUGETLB flag, then irrespective of the hugectl prefix or those environment variable, your program will consume huge pages for heap memory if available.
  • When you use MAP_HUGETLB, the mmap(2) call can fail (e.g. if your system does not have huge pages configured, or if some resource is exhausted), so you almost always should call mmap without MAP_HUGETLB as a fail back.

Are you using brk syscall?

  • If your program uses brk syscall directly to resize the heap then setting up those environment variables or hugectl will not have any impact on resized memory and resized memory will be backed by base pages instead of huge pages. That's because libhugetlbfs hijacks the morecore (called by malloc) to not to use brk syscall and back up those pages by huge pages instead but if you bypass morecore itself, then you'll be stuck with using base pages.

Sharing text section for multiple instances of same binary

  • By default, hugepages will load multiple copies of same program if it's instantiated multiple times. If you're running a binary for which you have to run multiple instances, you can save on text space by one of the following options:
  • Option 1: using environment variables
    • HUGETLB_SHARE=1
  • Option 2: using hugectl
    • Run the binary with prefix "hugectl --share-text"
  • Use with caution
    • Beware that with option you'll have to delete this shared text space when all instances of the programs are shut down. In our systems, when you use this option, text space will be backed up by a persistent file in /dev/hugepages/alflink-uid-XXXX (XXXX = uid of hte process using sharing) and we'll need to delete this file after the completion of the program. Please don't use this option if you have only one instance of the binary running at all the times which is usually the case for trade servers.

Using huge pages for stack?

  • It is not supported yet.

Can huge page have a negative impact?

  • there might be some cases in which they will have some negative impact. Consider 2 threads running on 2 different NUMA nodes, using huge pages increases the chances that both threads are using same page. When both threads are writing to same page, then page access for one of the thread will have to be from remote NUMA node instead of local NUMA node.
  • Using more memory than offered by huge pages. Please be warned that the HugePages pool must be sized appropriately. If the HugePages pool is exhausted and a process faults in a HugePage which has not yet been allocated the kernel will sent the process a SIGBUS.
  • 由于各种内存操作基本都要求按照page对齐，比如一个可执行文件映射到进程地址空间，根据文件大小的不同，平均算下来会浪费掉半个page size的物理内存，使用large page的话这个消耗就显得比较大了。
  • 系统运行一段时间后，会很难再也大块的连续物理内存，这时分配large page将会变的很困难，所以通常需要在系统初始化的时候就划分出一段物理内存给large page用（类似于DMA的内存分配），这样就减少了一些灵活性。
  • 动态large page（THP）在换出到外部的flash/disk和从flash/disk换入物理内存的过程会比normal size的page带来更大的开销（可参考这篇文章）。

Transparent huge page

自2.6.38版本开始支持THP。在应用需要huge page的时候，可通过memory compaction操作移动页面，以形成一个huge page，因为该过程不会被应用感知到，所以被称为"transparent"。

THP还只支持anonymous pages，而不支持page cache。

总的来说，目前THP优劣的平衡还存在一定的不确定性，在虚拟化的应用中，如果物理内存充足，通常建议使用单独划分的静态huge page，而不使用THP。
